# .env.example synced from master — single source of truth

## The rule

1. **Master is the only place a `.env.example` is hand-edited.** The
   canonical file lives at
   [`templates/.env.example`](../../templates/.env.example) in the
   master `chirag127/oriz` repo. ~65 keys today, plain `KEY=` lines,
   no comments.
2. **Every other repo's `.env.example` is a verbatim copy** generated
   by [`scripts/sync-env-example.sh`](../../scripts/sync-env-example.sh).
   That includes every site (`sites/oriz-*-site`), every browser /
   VS Code extension, every package (`packages/oriz-*`),
   `oriz-omnipost`, `oriz-lifestream`, every worker, every CLI.
3. **No hand-edits to a child `.env.example`.** Add / remove / rename
   a key on master, then run the sync script. A child repo's
   `.env.example` IS NOT a place to put repo-specific keys.
4. **CI enforces drift = failure.** Every PR (master + every repo)
   runs [`scripts/verify-env-example-sync.sh`](../../scripts/verify-env-example-sync.sh)
   (or its workflow equivalent) which `diff`s the repo's
   `.env.example` against `templates/.env.example`. Non-empty diff
   fails the PR. Per-repo deviation is forbidden.
5. **No comments inside any `.env.example`.** Carries forward from
   the prior rule and from the user's verbatim instruction. Just
   `KEY=` lines, one per line, in master template order.

## Why

- **One file to update, not 20+.** The family already has 11 sites,
  multiple packages, browser + VS Code extensions, `oriz-omnipost`,
  `oriz-lifestream`. Hand-editing every child for a new key (e.g.
  `WAKATIME_API_KEY` for the lifestream pipeline) is mechanical
  drudgery and a guaranteed source of drift.
- **Drift kills builds.** A site that's missing a key its CI step
  expects will fail mysteriously. Sync-from-master + CI diff means a
  drift IS the failure — caught at PR time, not 3 days later in a
  prod deploy.
- **Same superset everywhere = no archaeology.** A new contributor
  cloning any repo gets the full list. They never have to ask
  "which keys does THIS repo need?" — they fill in the values they
  need from [Doppler](../../services/secrets/doppler.md), leave the
  rest empty.
- **Pairs with org-level GitHub secrets.** Local dev reads the same
  key names that CI reads from
  [`chirag127`-org-level GH Actions secrets](./github-org-level-secrets.md).
  One naming surface, two delivery channels (local `.env` from
  Doppler; CI from org-level GH secrets).
- **User direction (verbatim) on 2026-06-20:** *"write the all teh
  env example files in the .env.example also so everything is
  managed from one project."* Per
  [`self-update-rule.md`](../agent/self-update-rule.md) +
  [`future-overrides-past.md`](../interaction/future-overrides-past.md), this is
  load-bearing.

## What this means concretely

- **Adding a new key.** Edit `templates/.env.example` on master,
  append the new `KEY=` line, run
  [`scripts/sync-env-example.sh`](../../scripts/sync-env-example.sh),
  commit + push the master repo and every touched submodule. See
  [`runbooks/sync-env-example-to-all-repos.md`](../../runbooks/operations/sync-env-example-to-all-repos.md).
- **Removing a key.** Same flow — master first, then sync. Removed
  keys also drop from the `chirag127` org-level GH secret list per
  [`github-org-level-secrets.md`](./github-org-level-secrets.md).
- **Drift detection.** CI step on every PR runs
  [`scripts/verify-env-example-sync.sh`](../../scripts/verify-env-example-sync.sh).
  Master CI checks every submodule's `.env.example`; each
  submodule's CI checks its own `.env.example` against the latest
  master `templates/.env.example` fetched via raw URL.
- **`.env` is gitignored.** Per
  [`no-hardcoded-secrets.md`](./no-hardcoded-secrets.md). Only
  `.env.example` is committed.
- **Tokens originate in Doppler.**
  `doppler secrets download --no-file --format env > .env`
  materialises a working local `.env` from the source of truth.
- **Empty values are normal.** A repo that doesn't actually consume
  a key still lists it with empty value — the contract is
  "key set matches master", not "every key is used at runtime".

## What this rule does NOT mean

- **Not** "the `.env` file itself is shared via git." It is not. Per
  [`no-hardcoded-secrets.md`](./no-hardcoded-secrets.md), no actual
  secret value enters any repo.
- **Not** "every repo USES every key at runtime." It only LISTS
  them; whether a key is read at runtime is the consumer's choice.
- **Not** an exemption from
  [`repos-work-independently.md`](../development/repos-work-independently.md).
  Each repo still must `pnpm install && pnpm build` after a
  `cp .env.example .env` (with empty values where unused).
- **Not** "GH Actions secrets are per-repo." They are
  [org-level](./github-org-level-secrets.md). The `.env.example`
  rule (this file) and the org-level secrets rule
  ([sibling](./github-org-level-secrets.md)) together implement the
  two-track model documented at
  [`decisions/security/env-and-secrets-single-source.md`](../../decisions/security/env-and-secrets-single-source.md).
