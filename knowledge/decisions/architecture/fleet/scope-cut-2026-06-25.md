# Scope-cut — 33 repos archived; only shipping content survives

## Decision

The [[build-gate-top3-must-have-defect]] rule applies at the **repo level**, not just the feature level. If a repo cannot describe what it ships to a user **today**, it is archived to `oriz-archive`. 33 repos were archived on 2026-06-25 under this rule. ~20 production-content repos survive.

This generalizes the build-gate from "every new feature must beat the top-3 results on a real defect" to "every repo must already be doing that for at least one user today." Scaffolds, placeholders, and "will-build-someday" stubs no longer count.

## Why now

Three signals lined up on 2026-06-25:

1. **Phase A reversal pressure** — 4 repos were created speculatively that morning. By afternoon none had shipping content. Keeping them would have set the precedent that scaffolds are okay.
2. **API-data deprecation** — 14 India-data APIs duplicate functionality already available from public 3rd-party APIs (api.mfapi.in, openexchangerates, weather APIs, IFSC datasets). Hosting + maintaining our own scrape adds cost with no user gain until we have an app that consumes them.
3. **Build-gate consistency** — [[build-gate-top3-must-have-defect]] already forbids shipping features that don't beat the top-3 alternatives. Applying it at the repo level closes the loophole where "the repo exists but the feature doesn't" was passing the bar.

## Implications

**Data-source cascade**: apps that previously consumed the in-house India-data APIs must swap to public 3rd-party APIs at build time:

- `oriz-janaushdhi-app` — PMBJP catalog data is downloadable directly from `janaushadhi.gov.in`; cache as JSON in repo per [[data-in-app-repos-not-separate]].
- `oriz-ncert-app` — NCERT PDFs hosted at `ncert.nic.in`; no API needed.
- `oriz-lore-app` — no API dependency; content authored in repo.

If a future tool surfaces (e.g. finance calculators), it consumes:

- Currency rates → `api.exchangerate-api.com` (free tier, no card).
- MF NAV → `api.mfapi.in` (direct, no card).
- Gold rates → scrape at build time via GH Actions if needed (no live API).

**Submodule count**: drops from 75 declared to ~20. `.gitmodules` must be swept. Re-verify with `cd /c/D/oriz && git submodule status | wc -l` after the cleanup commit.

**Family inventory file** ([[services/family-inventory]]) must be regenerated — every count in it is now stale.

## Future rule

**Same build-gate, applied at repo level — don't create a repo until you can describe what it ships THIS WEEK.** Concretely:

- No "scaffold" status. An app is either `production` (live URL with content/feature beating the top-3 alternative) or it does not yet exist.
- No speculative repos created in advance of a decision to draft. The decision-to-draft is the trigger for `gh repo create`, not the inverse.
- A repo that drops below the bar (e.g., bit-rotted content, stale data source, abandoned feature) re-enters the archive queue at the next fleet sweep.

The Phase A reversal on this date is the canonical example: 4 repos created hours earlier were archived the same day because they failed this rule.
